# MedArchive

**Кейс 2 — Автоматическая обработка архива прайсов клиник-партнёров**

Одной фразой: превращаем разнородный архив прайс-листов (PDF, скан, XLSX, DOCX) в **единую верифицированную базу услуг и цен** с поиском «кто оказывает услугу и по какой цене».

**Стек:** Python 3.11 · FastAPI · SQLAlchemy (SQLite dev / PostgreSQL prod) · pdfplumber · PyMuPDF + Tesseract (OCR rus+kaz+eng) · python-docx · openpyxl · RapidFuzz + sentence-transformers · React + Vite.

> Полная реализация ТЗ: парсинг всех форматов, нормализация, валидация §4.4, версионирование, REST API (OpenAPI) и админ-панель.

---

## Проблема и решение

**Проблема.** Прайсы партнёров приходят архивом в разных форматах и шаблонах: текстовые PDF, сканы (нужен OCR), Excel с произвольной структурой, Word с tracked changes. Названия услуг не унифицированы, валюты разные, у партнёра несколько прайсов на разные даты. Ручной разбор не масштабируется и даёт ошибки.

**Решение.** Конвейер, который автоматически:
- определяет формат и извлекает структуру (услуга, цена резидент/нерезидент);
- нормализует названия к целевому справочнику услуг;
- валидирует данные, ловит аномалии, конвертирует валюты в KZT;
- строит историю изменения цен (версионирование);
- отдаёт всё через REST API и веб-интерфейс поиска + админку.

**Ключевые возможности:** 4 формата + OCR · каскадная нормализация · 8 авто-проверок §4.4 · бессрочная история цен · очереди ручной разметки и верификации · дашборд качества.

## Архитектура

**Конвейер обработки:**

```
ZIP ─▶ распаковка ─▶ detect (формат) ─▶ очередь (PriceDocument.pending)
   ─▶ фоновый воркер:
        parse ─▶ normalize ─▶ validate ─▶ version ─▶ persist
   ─▶ БД ─▶ REST API ─▶ React UI (поиск / страница партнёра / админка / дашборд)
```

**Слои:** `parsers/` (detect, pdf_text, pdf_scan-OCR, xlsx, docx) · `normalization/` (matcher: exact → синонимы → fuzzy → эмбеддинги) · `validation/` (проверки §4.4, currency, versioning) · `ingestion/` (archive, partner-dedup, pipeline, worker) · `api/` · `frontend/`.

**Расширяемость (НФТ §5).** Ядро общается только с контрактами (`BaseParser`, `MatchResult`, `ValidationOutcome`). Новый формат/источник добавляется **регистрацией парсера в реестре** — без изменения ядра.

**Портативность.** UUID как `String(36)`, JSON через portable-тип → один код на SQLite (dev, zero-config) и PostgreSQL (prod, docker-compose). Очередь = строки `PriceDocument`, воркер на пуле потоков (масштабируется до Celery/RQ без изменения контракта).

## Возможности по ТЗ

| Требование ТЗ | Реализация |
| --- | --- |
| Парсинг PDF (текст) | pdfplumber/PyMuPDF, эвристики колонок услуга/цена |
| Скан-PDF (OCR) | PyMuPDF растеризация → Tesseract (rus+kaz+eng) → очистка артефактов |
| XLSX | openpyxl, обход листов, авто-поиск строки заголовков |
| DOCX (tracked changes) | принятие всех правок → финальный текст + таблицы |
| Нормализация §4.3 | каскад exact / синонимы / RapidFuzz / эмбеддинги; порог `MATCH_AUTO_THRESHOLD=0.85` |
| Валидация §4.4 | 8 проверок: цена>0, нерезидент≥резидент, пустое имя, дата в будущем, dedup, аномалия >50%, валюта, нет данных |
| Версионирование цен | при изменении старая версия архивируется (`is_active=False`), история бессрочно |
| Конвертация валют | не-KZT по курсу на дату прайса (`fx_rates.json`), оригинал сохраняется |
| Очереди | `unmatched` (ручная разметка) + `needs_review` (верификация) |
| REST API + админка | 13 эндпоинтов с OpenAPI/Swagger + React-панель |

## Демо-сценарий

**Запуск (SQLite, без Docker):**
```bash
cd backend
python3 -m venv .venv && . .venv/bin/activate
pip install -r requirements.txt
python -m scripts.bootstrap_demo        # справочник + демо-архив + обработка
uvicorn app.main:app --reload           # API + Swagger: http://localhost:8000/docs
cd ../frontend && npm install && npm run dev   # UI: http://localhost:5173
```

**Что показываем по шагам:**
1. **Загрузка архива** — админка «Загрузка», `POST /api/admin/upload` (ZIP).
2. **Статус обработки** — очередь документов (`done` / `needs_review`), `GET /api/admin/documents`.
3. **Дашборд** — документы, % нормализации, размеры очередей (`GET /api/admin/dashboard`).
4. **Поиск услуги** → список партнёров с ценами резидент/нерезидент (`GET /api/services/{id}/partners`).
5. **Страница партнёра** — полный прайс, контакты, дата актуальности.
6. **Очередь unmatched** — несопоставленные позиции с кандидатами (`GET /api/unmatched`).
7. **Ручной матч** — оператор выбирает услугу из справочника (`POST /api/match`).
8. **Верификация** — подтвердить/отклонить/исправить (`GET /api/admin/verification`, `POST /api/admin/verify`).

## Метрики качества

> Реальные результаты на демо-архиве (`scripts.quality_report`, см. `docs/REPORT.md`): 5 партнёров, 78 услуг справочника, 8 документов, 75 активных позиций.

**Нормализация:** **94.7%** авто-нормализации (71/75) при цели ТЗ ≥70% — ✅ ручных сопоставлений 0.

| Формат | Док. | Позиций | Сопоставлено | % |
| --- | --- | --- | --- | --- |
| docx | 2 | 15 | 15 | 100.0% |
| pdf | 3 | 21 | 20 | 95.2% |
| scan_pdf | 1 | 9 | 9 | 100.0% |
| xlsx | 2 | 30 | 27 | 90.0% |

**Производительность (НФТ §5):** текстовый документ макс. **0.10 с** (лимит 60 с) · OCR-скан **5.90 с** (лимит 180 с) — ✅ обе цели.

**Валидация §4.4:** флаги `NONRESIDENT_LT_RESIDENT` ×5, `INVALID_PRICE` ×3, `PRICE_ANOMALY` ×1.

**Версионирование** («Общий анализ крови»): v1 2025-01-15 = 2500 ₸ (архив) → v2 2024-01-10 = 2500 ₸ (архив) → **v3 2025-06-20 = 2750 ₸ (активна)**; 5 позиций с историей.

**Конвертация валют** (USD→KZT по курсу 505.0): Установка коронки 90 USD → 45 450 ₸ · Отбеливание зубов 80 USD → 40 400 ₸.

**Очереди:** `needs_review` = 10, `unmatched` = 3 (напр. «Криодеструкция папиллом азотом», уверенность 0.55).

## Соответствие критериям §8 и roadmap

| Критерий (вес) | Как закрыт |
| --- | --- |
| Качество извлечения (30%) | 4 формата + OCR + tracked changes; 8/8 документов обработано, 95.2% даже на pdf |
| Нормализация (25%) | **94.7%** авто (цель 70%); каскад + очередь unmatched с кандидатами |
| Валидация/верификация (20%) | 8 авто-проверок §4.4, аномалии >50%, бессрочное версионирование, очередь верификации |
| API и поиск (15%) | 13 REST-эндпоинтов, OpenAPI/Swagger на `/docs`, полнотекстовый `/search` |
| UX админки (10%) | загрузка, статусы, дашборд, очереди unmatched/верификации, ручной матч |

**Что дальше (roadmap):**
- мультиязычные эмбеддинги по флагу для «серой зоны» сопоставления;
- очередь на Celery/Redis вместо пула потоков для горизонтального масштаба;
- полнотекстовый поиск на Elasticsearch / MeiliSearch при росте объёма;
- расширение реестра парсеров под новые источники без изменения ядра.
